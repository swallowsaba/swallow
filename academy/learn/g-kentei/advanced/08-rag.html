章20 — RAG（検索拡張生成）Chapter 20 — Retrieval-Augmented Generation

LLM には 3 つの致命的弱点があります。(1) 知識カットオフ（学習データ以降の情報を知らない）、(2) 幻覚（自信を持って嘘を生成）、(3) 私的データ非対応。これらをすべて解決するのが RAG（Retrieval-Augmented Generation）── 質問を受けたら関連文書を検索して LLM に渡す、という単純で強力なアプローチ。本章では Embedding、Vector DB、Chunking、Reranking、GraphRAG、HyDE までを5部構成で扱います。

G-KENTEI · ADVANCED · CHAPTER 20

幻覚を、検索で消す。
LLMに、最新知識を。

RAG の原理は驚くほど単純：『質問が来たら、関連する文書を検索 → 文書と質問を一緒に LLM に渡す → 答えさせる』。これだけで、ファインチューニング不要・最新情報対応・自社データ活用が可能に。Lewis et al. 2020 で提案されて以来、エンタープライズ AI の事実上の標準となりました。

SECTION ONE

基本

RAGの動機、Embedding、Vector DB、Retrieval、Chunking、Rerank、Generation、評価、Hybrid、RAG vs Fine-tuneの10節。

基本

§1.1 RAGの動機 — LLMの3つの弱点

問題RAGによる解決
知識カットオフ（GPT-4は2023/4まで）最新文書を Index に追加するだけ
幻覚（Hallucination）事実を文書から引用させる
私的データ未対応社内文書を Vector DB に
引用元不明使った文書を明示
ドメイン特化困難専門文書だけ Index 化

§1.2 Embedding — テキストをベクトルに

文書とクエリを同じ意味空間の密ベクトル（典型 384〜3072次元）に変換。意味的に類似なものは近い距離に。代表モデル：

モデル次元特徴
OpenAI text-embedding-3-small1536API、安価、多言語
OpenAI text-embedding-3-large3072API、高精度
Cohere Embed v31024多言語、Re-rank と組み合わせ
Voyage AI1024〜専門ドメイン特化
BGE-large (BAAI)1024オープン、MTEBで上位
E5-mistral-7B4096LLM ベース、高精度
Sentence-Transformers (all-MiniLM-L6-v2)384軽量、CPU動作可

§1.3 Vector Database

大量のベクトルから類似ベクトルを高速検索するDB。ANN（Approximate Nearest Neighbor）アルゴリズムが核。

DB方式特徴
FAISSライブラリMeta、GPU対応、最高速
PineconeマネージドSaaS、運用簡単、商用
WeaviateOSSGraphQL、ハイブリッド検索
QdrantOSS（Rust）高速、フィルタリング強
ChromaDBOSSPython ネイティブ、開発向け
MilvusOSS大規模、Zilliz商用版
pgvectorPostgreSQL拡張既存DBに組込可

アルゴリズム：HNSW（Hierarchical Navigable Small World）が現代標準、IVF-PQ も実用。

§1.4 Retrieval — 類似度検索

Cosine 類似度　 sim(q, d) = (q·d) / (||q|| ||d||)
Dot Product　 q·d　 (正規化済みなら Cosine と同じ)
BM25　 古典的疎ベクトル（TF-IDF系）、キーワード一致

top-k 検索　 上位 k=5〜20 文書を返す。最近は k=50→Rerank で k=5 が定石
FIG 20.1 ／ RAG パイプライン。質問を埋め込みベクトル化 → Vector DBで類似検索 → 取得文書を LLM に渡して回答生成。

§1.5 Chunking 戦略

長い文書を埋め込む単位に分割：

  • Fixed-size：典型 200〜800 トークン、オーバーラップ 10〜20%
  • Semantic chunking：意味的境界（段落、見出し）で分割
  • Recursive：階層的に分割（LangChain RecursiveCharacterTextSplitter）
  • Late chunking（2024）：長文を埋め込んでから分割（コンテキスト保持）
  • Parent-Child：小チャンクで検索、親チャンクを渡す

§1.6 Reranking — 検索結果の再順序付け

Embedding ベース検索は速いが精度に限界。Cross-Encoder で top-50 を厳密採点 → top-5 に絞る2段階が標準。

モデル特徴
Cohere RerankAPI、多言語、商用最強
BGE RerankerOSS、BAAI、高精度
ColBERT遅延相互作用、効率と精度両立
monoT5/RankT5T5 ベース、CPU可

§1.7 Generation — プロンプト構築

プロンプト = システム指示 + 取得文書（複数） + 質問

典型構造
　 「以下の文脈を参考にして質問に答えなさい。
　 　文脈：<doc1><doc2><doc3>
　 　質問：<query>
　 　文脈に書かれていない事実は答えないこと」

§1.8 評価指標

  • Retrieval metrics：Recall@k、MRR、NDCG@k
  • Generation metrics：BLEU、ROUGE、BERTScore
  • RAG specific：Faithfulness（回答が文脈に忠実か）、Answer Relevance、Context Precision
  • フレームワーク：RAGAS、TruLens、ARES、LLM-as-Judge（GPT-4による自動評価）

§1.9 Hybrid Search — Dense + Sparse

Embedding（Dense、意味類似）と BM25（Sparse、キーワード一致）を組み合わせて Reciprocal Rank Fusion（RRF）で統合。固有名詞・略語・新語に強くなる。

§1.10 RAG vs Fine-tuning

判断基準RAGFine-tuning
知識の更新○ Index 追加のみ× 再学習
引用元の明示○×
スタイル・行動の変更△○
初期コスト低高
幻覚抑制○△
レイテンシ検索分加算低い

実用では両者の併用が増えている：Domain Adaptation で Fine-tune → RAG で最新情報。

SECTION TWO

応用

GraphRAG、Self-RAG、HyDE、Multimodal RAGの4節。

応用

§2.1 GraphRAG（Microsoft 2024）

文書からエンティティ・関係を抽出し知識グラフを構築。グラフのコミュニティ単位で要約を作成し、Global query（『全体の傾向は？』）にも対応。論文・企業文書の包括理解に強い。

§2.2 Self-RAG（Asai 2023）

LLM 自身が『検索が必要か』『取得文書が関連か』『回答が裏付けされているか』を Reflection Token で判断する自律的RAG。不必要な検索の削減と品質向上を両立。

§2.3 HyDE — Hypothetical Document Embeddings

Gao 2022。質問をそのまま埋め込む代わりに、LLM に仮の回答を生成させ、それを埋め込んで検索。質問と回答は意味空間で異なる位置にあり得るため、検索精度が向上することがある。

§2.4 Multimodal RAG

画像、表、グラフを含む PDF・スライドから情報抽出。CLIP 系統の画像-テキスト共通埋め込み、またはVision LLM（GPT-4V、Claude 3、Gemini）で画像チャンクを直接処理。技術文書・財務報告書解析で重要。

SECTION THREE

ハンズオン

LangChain、LlamaIndex、ChromaDB、Pinecone、Cohere Rerankの5節。

ハンズオン

§3.1 LangChain での最小 RAG

from langchain_community.document_loaders import PyPDFLoader from langchain.text_splitter import RecursiveCharacterTextSplitter from langchain_openai import OpenAIEmbeddings, ChatOpenAI from langchain_community.vectorstores import Chroma from langchain.chains import RetrievalQA # 1. 文書をロード＆分割 docs = PyPDFLoader('manual.pdf').load() splitter = RecursiveCharacterTextSplitter(chunk_size=500, chunk_overlap=50) chunks = splitter.split_documents(docs) # 2. Embedding + Vector DB embeddings = OpenAIEmbeddings(model='text-embedding-3-small') vectordb = Chroma.from_documents(chunks, embeddings, persist_directory='./db') retriever = vectordb.as_retriever(search_kwargs={'k': 5}) # 3. RAG チェーン qa = RetrievalQA.from_chain_type( llm=ChatOpenAI(model='gpt-4', temperature=0), retriever=retriever, return_source_documents=True) result = qa.invoke({'query': '返品ポリシーは？'}) print(result['result']) print("参照元:", [d.metadata for d in result['source_documents']])

§3.2 LlamaIndex

from llama_index.core import VectorStoreIndex, SimpleDirectoryReader from llama_index.llms.openai import OpenAI documents = SimpleDirectoryReader('./docs').load_data() index = VectorStoreIndex.from_documents(documents) query_engine = index.as_query_engine(llm=OpenAI(model='gpt-4o-mini')) response = query_engine.query('AI戦略の主要な柱は？') print(response) print("ソース:", response.source_nodes)

§3.3 ChromaDB スクラッチ

import chromadb from chromadb.utils import embedding_functions client = chromadb.PersistentClient(path='./chroma') ef = embedding_functions.OpenAIEmbeddingFunction( model_name='text-embedding-3-small') col = client.get_or_create_collection(name='docs', embedding_function=ef) # 追加 col.add(documents=['返品は購入後30日以内', '配送は3-5営業日'], ids=['1', '2'], metadatas=[{'cat':'return'}, {'cat':'shipping'}]) # 検索 results = col.query(query_texts=['返品方法'], n_results=2) print(results)

§3.4 Pinecone

from pinecone import Pinecone, ServerlessSpec pc = Pinecone(api_key='YOUR_KEY') pc.create_index( name='docs', dimension=1536, metric='cosine', spec=ServerlessSpec(cloud='aws', region='us-east-1')) idx = pc.Index('docs') idx.upsert(vectors=[ ('1', [0.1]*1536, {'text': '返品ポリシー...'}), ]) results = idx.query(vector=[0.1]*1536, top_k=5, include_metadata=True)

§3.5 Cohere Rerank で精度向上

import cohere co = cohere.Client('YOUR_KEY') # 1. Vector DB で top-50 取得 candidates = vectordb.similarity_search(query, k=50) docs = [d.page_content for d in candidates] # 2. Rerank で top-5 に絞る rerank = co.rerank(model='rerank-multilingual-v3.0', query=query, documents=docs, top_n=5) top_docs = [docs[r.index] for r in rerank.results] # 3. LLM に渡して回答生成 prompt = f"以下の文脈を参照: {top_docs}\n質問: {query}" answer = llm.invoke(prompt)
SECTION FOUR

シミュレーション

Chunk size、Rerank効果、Hybrid Searchの3節。

シミュレーション

SIM 1 — Chunk Size が精度に与える影響

>> 同じ文書セット、Chunk size を変えて Recall@5 評価

Chunk 100 tokens:　 Recall@5 = 0.62 (細切れで文脈不足)
Chunk 300 tokens:　 Recall@5 = 0.78
Chunk 500 tokens:　 Recall@5 = 0.84 ← スイートスポット
Chunk 1000 tokens: Recall@5 = 0.81 (1チャンクに多話題)
Chunk 2000 tokens: Recall@5 = 0.75

>> Overlap 50〜100 tokens で境界問題を緩和
>> Semantic chunking で更に向上の場合も

SIM 2 — Rerank の効果

>> FAQ データセット、Embedding のみ vs +Rerank

[Embedding (top-5)]
　 Recall@5 = 0.72, MRR = 0.58
　 レイテンシ: 50ms

[Embedding top-50 + Cohere Rerank top-5]
　 Recall@5 = 0.91, MRR = 0.82
　 レイテンシ: 250ms (+200ms)

>> 2段階で精度大幅向上、レイテンシは少し増加
>> 多くの本番システムで採用される定石パターン

SIM 3 — Hybrid Search（Dense + Sparse）

>> 製品マニュアル、固有名詞・型番含む

[Dense (Embedding)]
　 一般的な質問: Recall@5 = 0.85
　 型番含む質問:　 Recall@5 = 0.42 ← 弱点

[Sparse (BM25)]
　 一般的な質問: Recall@5 = 0.61
　 型番含む質問:　 Recall@5 = 0.94

[Hybrid (RRF統合)]
　 一般的な質問: Recall@5 = 0.91
　 型番含む質問:　 Recall@5 = 0.89

>> Hybrid で両方の強みを取り込む
SECTION FIVE

問題

G検定形式 20問。

問題

PROBLEM 1
RAG が解決する LLM の主な弱点は。
① 計算速度　 ② 知識カットオフ、幻覚、私的データ非対応　 ③ パラメータ数　 ④ メモリ
ANSWER ② LLM の学習データ以降の知識・自社データ・幻覚（誤情報生成）を、関連文書検索で補完。
PROBLEM 2
RAG パイプラインの基本構成は。
① クラスタ→分類　 ② Embed → Vector DB 検索 → 文脈付き LLM プロンプト → 回答　 ③ Fine-tune のみ　 ④ ランダム
ANSWER ② Lewis 2020。質問を埋め込み、Vector DB から類似文書取得、LLM に文書+質問を渡して回答生成。
PROBLEM 3
Embedding モデルの代表として誤りはどれか。
① text-embedding-3-large (OpenAI)　 ② BGE-large (BAAI)　 ③ Cohere Embed v3　 ④ XGBoost
ANSWER ④ XGBoost は勾配ブースティング、テキスト埋め込み用ではない。MTEB ベンチマークで競合中。
PROBLEM 4
Vector Database で標準的な高速検索アルゴリズムは。
① ブルートフォース　 ② HNSW（Hierarchical Navigable Small World）、IVF-PQ　 ③ K-means　 ④ Decision Tree
ANSWER ② ANN（Approximate Nearest Neighbor）の代表。HNSW は graph-based で精度・速度両立。FAISS, Pinecone, Qdrant, Weaviate で実装。
PROBLEM 5
Chunking で典型的なオーバーラップ率は。
① 0%　 ② 10〜20%（境界問題緩和）　 ③ 50%以上　 ④ 100%
ANSWER ② オーバーラップで文脈の境界での情報損失を緩和。chunk_size=500 なら overlap=50〜100 が典型。
PROBLEM 6
Reranking の目的は。
① 計算高速化　 ② Embedding 検索で得た上位文書を Cross-Encoder で再評価し精度向上　 ③ 学習　 ④ 正則化
ANSWER ② 2段階パイプライン：Embedding で top-50 → Rerank で top-5。Cohere Rerank, BGE Reranker が代表。
PROBLEM 7
Hybrid Search の典型構成は。
① Dense のみ　 ② Embedding（Dense）+ BM25（Sparse）を RRF で統合　 ③ ランダム　 ④ Sparse のみ
ANSWER ② Reciprocal Rank Fusion（RRF）で両者の順位を融合。固有名詞・型番に強い BM25 と意味類似の Dense を統合。
PROBLEM 8
RAG vs Fine-tuning の使い分けで正しいのは。
① RAG = スタイル変更　 ② 知識追加・更新は RAG、行動・スタイル変更は Fine-tune　 ③ どちらも同じ　 ④ Fine-tune が常に上
ANSWER ② RAG = 知識（事実）、Fine-tune = 振舞い（口調、フォーマット）。実用では併用が多い。
PROBLEM 9
『Faithfulness』が評価する対象は。
① 速度　 ② 生成回答が取得文書に忠実か（捏造でないか）　 ③ パラメータ数　 ④ コスト
ANSWER ② RAGAS, TruLensなどで自動評価。LLM-as-Judge（GPT-4） が一般的。幻覚抑制の指標。
PROBLEM 10
GraphRAG（Microsoft 2024）の特徴は。
① 単純な BM25　 ② 文書から知識グラフを構築、Global query（全体傾向）にも対応　 ③ Fine-tune必須　 ④ クラスタリング
ANSWER ② エンティティ・関係抽出 → グラフ → コミュニティ要約。『この文書集合全体の傾向は』のような Global question に強い。
PROBLEM 11
HyDE（Hypothetical Document Embeddings）の手順は。
① 質問を直接埋め込み　 ② LLMに仮回答を生成させ、それを埋め込んで検索　 ③ 教師ありのみ　 ④ クラスタ
ANSWER ② Gao 2022。質問と回答は意味空間で異なる位置にあり得るため、仮回答を介して検索精度向上。
PROBLEM 12
Self-RAG が用いる仕組みは。
① ランダム検索　 ② Reflection Token で『検索が必要か』『文書が関連か』『裏付けあるか』を LLM 自身が判断　 ③ クラスタリング　 ④ 教師あり
ANSWER ② Asai 2023。不要検索の削減と品質向上を両立。LLMが検索の必要性を判断。
PROBLEM 13
Cosine類似度の式は。
① q + d　 ② (q·d)/(||q|| ||d||)　 ③ |q - d|　 ④ q × d
ANSWER ② ベクトル間の角度を測る。長さ非依存で意味の類似度に適する。RAGの標準距離。
PROBLEM 14
RAG の典型的な失敗パターンは。
① 速度向上　 ② Chunk が大きすぎる/小さすぎる、Embedding がドメイン外、適切な Rerank不在　 ③ パラメータ削減　 ④ 学習率
ANSWER ② 実用 RAG は『調整勝負』。Chunk size、Embedding 選択、Rerank、プロンプト構造、メタデータフィルタなどの組合せ最適化が肝。
PROBLEM 15
Multimodal RAG が扱う主なデータは。
① テキストのみ　 ② 画像、表、グラフを含む PDF・スライド（CLIP系または Vision LLM 利用）　 ③ 音声のみ　 ④ コード
ANSWER ② 技術文書・財務報告書は画像・表が多い。GPT-4V, Claude 3, Gemini が直接処理可能、または CLIP 系で画像埋め込み。
PROBLEM 16
LangChain の役割は。
① 学習　 ② LLMアプリ構築の統合フレームワーク（RAG、Agent、チェーン）　 ③ Vector DB のみ　 ④ Embedding
ANSWER ② 文書ローダー、Splitter、Vector DB、LLM、Retriever、Agent を統合。LlamaIndex は RAG 特化の競合。
PROBLEM 17
Sentence-Transformers の代表モデルは。
① BERT base のみ　 ② all-MiniLM-L6-v2（軽量384次元、CPU動作可）　 ③ GPT-4　 ④ ResNet
ANSWER ② 開発・プロトタイプの定番。BGE, E5, GTE などより新しいモデルもオープン提供。
PROBLEM 18
RAGAS フレームワークが提供する RAG 評価指標は。
① Accuracy のみ　 ② Faithfulness, Answer Relevance, Context Precision, Context Recall　 ③ MSE　 ④ R²
ANSWER ② LLM-as-Judge で自動評価。RAG 特有の側面（事実忠実性、文脈の必要性・十分性）を測定。
PROBLEM 19
Vector DB の選択基準として誤りはどれか。
① 規模・QPS要件　 ② フィルタリング機能　 ③ ホスティング（SaaS vs Self-host）　 ④ どれも同じ性能
ANSWER ④ Pinecone（マネージド）、Weaviate（OSS+クラウド）、Qdrant（OSS Rust高速）、Chroma（軽量）、Milvus（大規模）など特性は大きく違う。
PROBLEM 20
プロダクション RAG での重要な工夫として誤りはどれか。
① クエリ書き換え（query rewriting）　 ② Rerank　 ③ メタデータフィルタ　 ④ 必ず Chunk size を 100 にする
ANSWER ④ Chunk size はデータ・タスク依存で最適化対象。100 だと文脈不足の場合が多い。500前後が経験的に良いが、用途で異なる。
