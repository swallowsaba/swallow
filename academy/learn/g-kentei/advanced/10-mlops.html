G検定 応用編 — 章34

モデルを動かし続ける。
MLの本番運用学。

研究と本番運用は別物。研究では『精度が高ければ良い』ですが、本番では『常に安定動作・障害時の迅速復旧・データドリフト対応・複数モデルA/B比較・再現性保証』が求められます。MLOps は DevOps を MLに拡張した分野で、企業導入の成功率を左右する重要技術です。

SECTION ONE

基本

MLOps概要、データパイプライン、Feature Store、モデル管理、CI/CD、デプロイ、モニタリング、Drift、A/B、Canaryの10節。

基本

§1.1 MLOps とは — ML × DevOps

機械学習システムのライフサイクル（データ収集→学習→評価→デプロイ→運用→再学習）全体を自動化・標準化する分野。Google『Hidden Technical Debt in ML Systems』(2015) が原点的論文。

FIG 22.1 ／ MLOps ライフサイクル。データ収集→学習→評価→デプロイ→運用、Drift検知時に自動再学習でループ。

§1.2 データパイプライン

本番データをETL（Extract Transform Load）で学習可能な形式に。Apache Airflow（DAG ベースのスケジューラ）、dbt（SQL中心）、Dagster、Prefect 等。データ品質チェック（Great Expectations）も統合。

§1.3 Feature Store

特徴量を再利用可能に管理。同じ特徴量を学習時とサービング時で一貫して提供（Training-Serving Skew防止）。Feast（OSS）、Tecton、AWS SageMaker Feature Store、Vertex AI Feature Store。

§1.4 モデル管理（Model Registry）

学習したモデルをバージョン管理。MLflow Model Registry、Weights & Biases、Neptune、DVC + Git-LFS。実験追跡（パラメータ、メトリクス、アーティファクト）と統合。

§1.5 CI/CD for ML（CT 含む）

段階内容
CI（Continuous Integration）コード・データ変更で自動テスト
CD（Continuous Deployment）合格モデルを自動本番デプロイ
CT（Continuous Training）Drift検知やスケジュールで自動再学習

§1.6 モデルデプロイ方式

方式用途例
REST API同期推論FastAPI, Flask, KServe
gRPC低レイテンシTriton Inference Server
バッチ推論非同期、大量Spark, AWS Batch
Edge推論デバイス上TF Lite, ONNX, Core ML
Serverless低頻度、トラフィック変動大AWS Lambda, Cloud Functions

§1.7 モニタリング — 本番運用の眼

  • システムメトリクス：レイテンシ、スループット、エラー率、CPU/GPU使用率 (Prometheus + Grafana)
  • モデルメトリクス：予測分布、入力分布、ラベル分布、精度（ラベル取得後）
  • Data Drift：本番入力分布の変化（Evidently、Alibi Detect）
  • Concept Drift：入力-出力関係の変化

§1.8 Drift検知

Data Drift（共変量シフト）　 P(X) が変化、 P(Y|X) は不変
Concept Drift　 P(Y|X) が変化（季節要因、トレンド）
Label Drift　 P(Y) が変化（クラス頻度変動）

検知手法：KS検定、PSI（Population Stability Index）、Wasserstein距離、Mahalanobis距離

§1.9 A/B Testing と Multi-Armed Bandit

A/B Test：本番トラフィックを2モデルに均等に分割、統計検定で勝者決定。Multi-Armed Bandit：効果の高いモデルに動的にトラフィックを偏らせる（Thompson Sampling、UCB）。

§1.10 Canary Release と Shadow Mode

Canary：新モデルを少量トラフィック（1〜5%）に流して安全確認、段階的に拡大。Shadow Mode：本番トラフィックを両方に流し、新モデルは予測のみ記録（影響なし）、安全に評価。

SECTION TWO

応用

SageMaker、Vertex AI、Kubeflow、MLflow vs DVC の4節。

応用

§2.1 AWS SageMaker

マネージドML プラットフォーム。Studio (IDE)、Training Jobs（マネージドGPU）、Endpoints（推論）、Model Registry、Feature Store、Pipelines（ワークフロー）、Clarify（Bias検知）、Ground Truth（ラベル付け）の統合。エンタープライズで広く採用。

§2.2 Google Vertex AI

Google Cloud の統合MLプラットフォーム。Workbench（Jupyter）、Custom Training、Online Prediction、Model Registry、Pipelines（Kubeflow ベース）、Feature Store、Model Monitoring、Explainable AI 統合。Gemini モデル等への直接アクセスも。

§2.3 Kubeflow — Kubernetes 上の ML

Kubernetes ネイティブの OSS MLプラットフォーム。Notebooks、Pipelines（DAG）、KServe（推論サーバ）、Training Operator（分散学習）、Katib（HPO）。クラウド非依存、マルチクラウドや on-premise で人気。

§2.4 ツール比較

ツール主な役割
MLflow実験追跡、モデル管理、デプロイ統合（OSS、ベンダー非依存）
Weights & Biases実験追跡＋ハイパラ最適化（クラウド SaaS）
DVCデータ・モデルバージョン管理（Git拡張）
Neptune実験追跡（SaaS）
ClearMLパイプライン＋実験追跡（OSS）

ハンズオン

MLflow、FastAPI、Docker、Prometheus、Drift検出の5節。

ハンズオン

§3.1 MLflow で実験追跡

import mlflow from sklearn.ensemble import RandomForestClassifier mlflow.set_experiment("churn_prediction") with mlflow.start_run(): params = {"n_estimators": 100, "max_depth": 10} mlflow.log_params(params) model = RandomForestClassifier(**params).fit(X_tr, y_tr) acc = model.score(X_te, y_te) mlflow.log_metric("accuracy", acc) mlflow.sklearn.log_model(model, "model") # 後で UI（http://localhost:5000）で全実験を比較

§3.2 FastAPI で推論サーバ

from fastapi import FastAPI from pydantic import BaseModel import mlflow.pyfunc app = FastAPI() model = mlflow.pyfunc.load_model("models:/churn/Production") class Features(BaseModel): age: int tenure: float monthly_charges: float @app.post("/predict") def predict(f: Features): pred = model.predict([[f.age, f.tenure, f.monthly_charges]]) return {"churn_probability": float(pred[0])} # uvicorn main:app --reload で起動

§3.5 Drift検出（Evidently）

from evidently.report import Report from evidently.metric_preset import DataDriftPreset ref = pd.read_csv("reference.csv")　 # 学習時データ cur = pd.read_csv("production.csv") # 本番データ report = Report(metrics=[DataDriftPreset()]) report.run(reference_data=ref, current_data=cur) report.save_html("drift_report.html") # PSI > 0.2 で警告、各特徴量の分布変化を可視化

シミュレーション

モデルバージョン管理、Drift検知、Canary の3節。

シミュレーション

SIM 1 — モデルバージョン管理

>> MLflow Model Registry での運用例

[2025-01-15] v1 (acc 0.84) → Staging
[2025-01-20] v2 (acc 0.87) → Staging
[2025-01-22] v2 → Production（v1は Archive へ）
[2025-02-10] Data Drift 検知（PSI=0.34）
[2025-02-12] v3（再学習、acc 0.89）→ Staging
[2025-02-15] Canary 5% v3 → 24時間問題なし
[2025-02-16] Canary 50% v3 → 24時間OK
[2025-02-17] v3 → Production（v2は Archive）

>> 各バージョンは MLflow で再現可能
>> ロールバックも 1コマンドで可能

SIM 2 — Drift 検知の実際

>> e-commerce 推薦モデル、6ヶ月運用

Month 1: PSI=0.05, acc=87% （安定）
Month 2: PSI=0.08, acc=86%
Month 3: PSI=0.13, acc=84% （Holiday Season）
Month 4: PSI=0.21, acc=80% ← Drift閾値超過
Month 5: 再学習実施 → acc=88% に回復
Month 6: PSI=0.06, acc=87% （安定）

>> Drift検知 → 再学習トリガで性能維持
>> PSI 0.2 が一般的な警告閾値

SIM 3 — Canary Release

>> 新モデル v3 を Canary でロールアウト

[Hour 0] v3: 1% traffic, v2: 99%
  v3 latency=120ms, v2=125ms, v3 error=0.1%
[Hour 6] v3: 5%
[Hour 24] v3: 25%, パフォーマンス問題なし
[Hour 48] v3: 50%
[Hour 72] v3: 100% (Full rollout)

>> 異常時は数秒で v2 にロールバック可能

SECTION FIVE

問題

G検定形式 20問。

問題

PROBLEM 1
MLOps の主な対象は。
① モデル学習のみ　 ② MLシステムのライフサイクル全体（データ・学習・デプロイ・運用・再学習）の自動化　 ③ データ収集のみ　 ④ 評価のみ
ANSWER ② Google『Hidden Technical Debt in ML Systems』が原点。コードはML全体の5〜10%、残りは MLOps インフラ。
PROBLEM 2
Feature Store の主な目的は。
① モデル管理　 ② 学習時とサービング時で一貫した特徴量提供、Training-Serving Skew 防止　 ③ パラメータ追跡　 ④ ログ収集
ANSWER ② Feast、Tecton、SageMaker Feature Store、Vertex AI Feature Store。特徴量の再利用と一貫性が核心。
PROBLEM 3
MLflow の主機能でないのは。
① 実験追跡（パラメータ、メトリクス）　 ② モデルレジストリ　 ③ プロジェクトパッケージング　 ④ データベース管理
ANSWER ④ MLflow は Tracking、Models、Projects、Registry の4機能。データベース自体は管理対象外。
PROBLEM 4
CI/CD for ML で『CT』が表すのは。
① Continuous Testing　 ② Continuous Training（Drift時等の自動再学習）　 ③ Continuous Tracking　 ④ Continuous Tagging
ANSWER ② MLOps の独自概念。CI（コードテスト）+ CD（デプロイ）+ CT（再学習）の3要素。
PROBLEM 5
Data Drift と Concept Drift の違いは。
① 同じ　 ② Data: P(X) 変化、Concept: P(Y|X) 変化　 ③ Drift は無関係　 ④ ラベルだけ
ANSWER ② Data Drift = 入力分布の変化（季節、ユーザー層変化）、Concept Drift = 入力→出力関係の変化（嗜好変化）。両方を監視。
PROBLEM 6
PSI（Population Stability Index）の用途は。
① 学習　 ② 分布変化の定量化（金融でDrift検知に標準）。0.2以上で警告　 ③ パラメータ調整　 ④ ノイズ生成
ANSWER ② PSI = Σ (P_new - P_ref) · ln(P_new / P_ref)。0.1未満=安定、0.1-0.2=やや変化、0.2以上=大きな変化。
PROBLEM 7
Canary Release の特徴は。
① 全トラフィック切替　 ② 新モデルに少量トラフィック（1〜5%）流して段階的に拡大、異常時即ロールバック　 ③ シャドウ　 ④ A/B均等
ANSWER ② Kubernetes/Istio で実装。安全なロールアウトの定番。Blue-Green デプロイより漸進的。
PROBLEM 8
Shadow Mode の特徴は。
① ユーザに新モデル結果を返す　 ② 本番トラフィックを両方に流すが新モデル結果は記録のみ、ユーザ影響ゼロで評価　 ③ ランダム　 ④ オフライン
ANSWER ② 本番影響なしで新モデルを評価。レイテンシ、メモリ、予測一致度を確認。Canary 前の安全策。
PROBLEM 9
Kubeflow の主な特徴は。
① クラウド専用　 ② Kubernetesネイティブの OSS、MLパイプライン・分散学習・推論を統合　 ③ AWS専用　 ④ Python専用
ANSWER ② Kubeflow Pipelines, KServe, Training Operator, Katib (HPO)。クラウド非依存、on-premise でも動く。
PROBLEM 10
AWS SageMaker の主機能でないのは。
① マネージドGPU 学習　 ② Endpoint デプロイ　 ③ Feature Store　 ④ 仮想通貨マイニング
ANSWER ④ SageMaker は ML特化マネージドサービス。Studio、Training Jobs、Endpoints、Pipelines、Clarify（Bias）、Ground Truth など。
PROBLEM 11
バッチ推論が向くケースは。
① リアルタイム必要　 ② 大量データを定期的に処理（夜間バッチ、レポート、推薦の事前計算）　 ③ チャット　 ④ ストリーミング
ANSWER ② Spark、AWS Batch、Airflow で実装。リアルタイム性不要・GPUコスト最適化が目的。
PROBLEM 12
Edge推論の典型ユースケースは。
① 大型サーバ　 ② スマホ・IoT・自動車などデバイス上で推論（TF Lite、ONNX、Core ML）　 ③ クラウドのみ　 ④ オフライン学習
ANSWER ② プライバシー・遅延・通信コストの観点でデバイス上推論が重要。量子化（INT8）、Pruning でモデル軽量化。
PROBLEM 13
A/B Test と Multi-Armed Bandit の違いは。
① 同じ　 ② A/B は固定割合、MAB は性能高いモデルに動的に割合増加（Thompson Sampling、UCB）　 ③ A/B はクラスタリング　 ④ MAB は強化学習
ANSWER ② MAB は Exploration-Exploitation を強化学習的に最適化。総合報酬最大化、A/B Test の発展形。
PROBLEM 14
Triton Inference Server の特徴は。
① 学習専用　 ② NVIDIA製、高性能推論サーバ、複数フレームワーク（PyTorch、TF、ONNX）・複数モデルを統合提供　 ③ DB　 ④ コンパイラ
ANSWER ② gRPC/HTTP両対応、Dynamic Batching、Model Concurrency でGPUを最大活用。SageMaker・Vertex でも統合。
PROBLEM 15
DVC の主機能は。
① ニューラルネット　 ② データ・モデルのバージョン管理（Gitに統合、大ファイルは外部ストレージへ）　 ③ 学習　 ④ クラウド管理
ANSWER ② Data Version Control。Git は大ファイルが苦手、DVC はメタ情報のみGit、本体はS3/GCS等。再現性確保に重要。
PROBLEM 16
Apache Airflow の主機能は。
① ML特化　 ② 汎用ワークフロー（DAG）スケジューラ、データパイプライン・ML パイプライン用　 ③ DBMS　 ④ Webサーバ
ANSWER ② Python で DAG を記述、スケジュール実行。MLOps では学習・推論・前処理を構成。代替に Dagster、Prefect、Argo。
PROBLEM 17
Training-Serving Skew の典型原因は。
① 学習データが少ない　 ② 学習時とサービング時で特徴量の計算ロジックが微妙に異なる　 ③ パラメータ多い　 ④ ランダム性
ANSWER ② 例：学習で平均値補完、サービングで0補完。Feature Store で計算ロジック共有することで防止。
PROBLEM 18
本番運用で監視すべきメトリクスでないのは。
① レイテンシ p99　 ② スループット（QPS）　 ③ エラー率　 ④ 開発者のメール頻度
ANSWER ④ システム面（レイテンシ、QPS、エラー率、CPU/GPU）とモデル面（予測分布、入力分布、Drift）を監視。Prometheus + Grafana が標準。
PROBLEM 19
Model Card の役割は。
① 学習データ　 ② モデルの用途、性能、制限、Bias情報を文書化（Google Mitchell 2019）。透明性確保　 ③ 名刺　 ④ クラスタリング
ANSWER ② モデルの『使用説明書』。EU AI Act、ISO/IEC 23053 などで義務化方向。Hugging Face で標準化。
PROBLEM 20
『MLOps Maturity』レベル0の特徴は。
① 完全自動化　 ② 全て手動、学習・デプロイ・モニタリング無連携　 ③ Drift自動再学習　 ④ MLOpsなし
ANSWER ② Google MLOps Maturity Model。Level 0=Manual、Level 1=ML Pipeline自動化、Level 2=CI/CD/CT完全統合。多くの企業はLevel 0-1。
